# weight-keeper — runbook

Сторож от отката веса. Демон на Ubuntu слушает ntfy, на каждое число отдаёт
зону, ведёт базу в формате Monitor Your Weight и журнал решений.

Документ написан под один принцип: **ты должен понимать и чинить эту штуку
без меня и без чтения Python.** Всё состояние — в текстовых файлах, вся
арифметика — на бумаге воспроизводима. Ниже — как.

---

## 1. Что где лежит

В каталоге бота (клон репозитория) и в `DATA_DIR` (если задан в `.env`):

| Файл | Что это | Кто правит |
|------|---------|-----------|
| `bot.py` | сеть, ntfy, главный цикл | код |
| `keeper.py` | вся логика: формат, EMA, храповик, зоны | код |
| `config.json` | пороги зон, периоды, диапазоны | ты, без знания Python |
| `.env` | секреты: юзер/пароль ntfy, URL, топики | ты, локально, `chmod 600` |
| `weight.txt` | **база** = бэкап в формате приложения | бот; ты — осторожно |
| `state.json` | состояние сторожа (пол, EMA, окно тренда) | бот; ты — для подъёма пола |
| `decisions.log` | летопись решений, append-only | только бот, ты читаешь |
| `weight-keeper.service` | systemd-юнит | ставится один раз |

`state.json.prev` — снимок для `undo`. `*.tmp` — мусор атомарной записи, можно игнорировать.

Технический лог демона (реконнекты, старт/стоп) — в journald, не в файле:
`journalctl -u weight-keeper -f`. `decisions.log` — это другое: дневник решений
(«опустил пол», «сменил зону», «вернулся после паузы»), его читаешь как историю.

## 2. Установка с нуля

```bash
# 1. На ntfy-сервере — пользователь под бота (least privilege)
sudo ntfy user add bot
sudo ntfy access bot priv_weight_in  read-only
sudo ntfy access bot priv_weight_out write-only

# 2. На хосте бота
git clone <repo-url> weight-keeper
cd weight-keeper
cp .env.example .env
nano .env            # заполнить URL, юзера/пароль бота, топики
chmod 600 .env

# 3. Положить базу и инициализировать состояние
#    (скопируй свой экспорт из приложения в weight.txt)
cp /path/to/backup.txt weight.txt
python3 bot.py --cold-start      # прочитает историю, поставит пол = текущая EMA

# 4. Демон под systemd
sudo cp weight-keeper.service /etc/systemd/system/
sudo nano /etc/systemd/system/weight-keeper.service   # поправить User и пути
sudo systemctl daemon-reload
sudo systemctl enable --now weight-keeper
journalctl -u weight-keeper -f                          # смотрим, что поднялся
```

Зависимостей нет — только системный Python 3 (нужен 3.9+). Ни pip, ни venv.

## 3. Как считается зона (чтобы проверять бота, а не верить ему)

Три величины, всё в `state.json` и `config.json`:

**EMA** — сглаженный вес. Каждое новое число тянет EMA на долю
`alpha = 1 − exp(−Δдней / tau)`, где `tau = tau_days` (по умолчанию 5).
Чем больше пропуск между взвешиваниями — тем сильнее новое число (старое
«выдыхается»). Поэтому дыры в данных не врут.

**Пол (floor)** — твой лучший сглаженный вес. Едет вниз сам: если EMA опустилась
ниже пола — пол = EMA (автоспуск, пишется в журнал). Вверх — только руками (раздел 5).

**Просадка** = `EMA − пол`. Насколько ты сейчас выше своего лучшего.

Зона по просадке (с гистерезисом `hysteresis_kg`, чтобы не мигало у границы):

```
просадка < yellow_threshold_kg            -> 🟢
yellow_threshold_kg ≤ просадка < red_...  -> 🟡
просадка ≥ red_threshold_kg               -> 🔴
```

Вторичный сигнал — **наклон** тренда: линейная регрессия EMA за последние
`slope_window_days` дней (окно лежит в `state.json` → `ema_window`), переведённая
в кг/неделю. Если наклон круче `slope_yellow`/`slope_red` — поднимает зону.

**Итоговая зона = худшая из двух** (храповик и наклон). В грейс-периоде после
паузы зона режется максимум до жёлтой.

Проверить руками: возьми `ema` и `floor` из `state.json`, вычти — получишь
просадку. Сравни с порогами из `config.json`. Это и есть зона. Никакой магии.

## 4. Повседневное (happy path и мелочи)

- **Взвесился** → шлёшь `64.7` в топик ввода → приходит зона. Всё.
- **Ошибся числом** → `undo` (откатит последнюю запись за сегодня), потом верное число.
- **Хочешь проверить статус** без записи → `status`.
- **Забыл, как команды** → `help` (пришлёт шпаргалку файлом).
- **Второе число за день** заменяет первое — одно взвешивание в день.

## 5. Подъём пола (трение осмысленное, не театральное)

Пол вверх = ты признаёшь новую норму выше прежней. Это уступка, и она должна
стоить усилия — не наказания, а **формулировки**. Ритуал:

```bash
ssh на_хост_бота
nano state.json
```

Поменять два поля:

```json
"floor": 66.0,
"floor_raise_reason": "почему поднимаю — своими словами, не короче 10 символов",
```

Сохранить. Рестарт **не нужен** — бот читает `state.json` с диска при следующем
взвешивании. На ближайшем числе он заметит, что пол вырос, и запишет в журнал:

- причина внятная → `пол поднят руками 64.7 -> 66.0; причина: ...`
- причина пустая/короткая → `пол поднят руками ... БЕЗ внятной причины`
  (пол всё равно примет — робастность важнее; но в летописи останется пометка).

Почему причина обязательна: в момент «да ладно, поем» написать вслух, *почему*
сдаёшься, — само по себе отрезвляет. Половина поводов отваливается, когда видишь
их буквами. Это трение, которое делает работу.

Поле `last_known_floor` бот ведёт сам — не трогай его. Именно по нему он
понимает, что пол подняли извне.

## 6. Пауза (отпуск без весов)

Ничего делать не надо. Вернулся — просто шли число как обычно.

Если перерыв больше `gap_days_vacation` дней (по умолчанию 4), бот:

- помечает ближайшие `grace_readings_after_gap` (3) числа предварительными;
- замораживает пол (не опускает на воде/обезвоживании);
- не даёт красную зону, пока данные не устаканятся.

В журнал попадёт `возврат после паузы N дн.`.

## 7. Бэкап базы

Раз в неделю (день = `backup_weekday`, 0 = понедельник) бот сам шлёт `weight.txt`
вложением в топик вывода — падает тебе в телефон. Это твоя страховка: умрёт VPS —
база (6,5 лет ручного труда) останется в истории ntfy и в телефоне.

Хочешь бэкап прямо сейчас — проще всего забрать файл руками:
`scp хост:~/weight-keeper/weight.txt .`

## 8. Troubleshooting

**Бот молчит на числа.**
```bash
systemctl status weight-keeper       # жив ли
journalctl -u weight-keeper -n 50     # последние логи
```
Частые причины: лежит ntfy-сервер; неверный юзер/пароль/URL в `.env`; топик в
`.env` не совпадает с тем, куда пишешь.

**Бот падает на старте сразу.**
Почти всегда — кривой `state.json` (поправил руками, забыл запятую/кавычку).
Мы выбрали «падать сразу и громко», так что в `journalctl` будет внятная строка:
либо `JSONDecodeError` (синтаксис), либо `state.json: поле 'floor' = ... вне
диапазона` (опечатка в числе, например пропущенная точка → 660 вместо 66.0).
Открой `state.json`, поправь, бот поднимется сам (`Restart=always`).

**Стрим всё время переподключается.**
В журнале `обрыв соединения с ntfy ... переподключение через Nс`. Если это
постоянно — проблема на стороне ntfy-сервера или сети между хостами, не у бота.
Бот сам наращивает паузу до 60с и продолжает пытаться.

**Хочу пересчитать состояние с нуля** (например, поправил историю в `weight.txt`):
```bash
sudo systemctl stop weight-keeper
python3 bot.py --cold-start      # перестроит state.json из weight.txt
sudo systemctl start weight-keeper
```

**Хочу сменить порог зоны.**
`nano config.json`, поменять число, сохранить. Знание Python не нужно. Изменение
подхватится на следующем взвешивании (демон читает конфиг при старте — если
менял пороги, сделай `systemctl restart weight-keeper`).

## 9. Тестовый прогон и вайп (тест на проде с последующим сбросом)

Поигрался на реальной базе — хочешь начать начисто:
```bash
sudo systemctl stop weight-keeper
rm state.json state.json.prev decisions.log     # weight.txt оставь!
python3 bot.py --cold-start
sudo systemctl start weight-keeper
```
База (`weight.txt`) при этом не страдает — сносим только производное состояние.
